1) High-Level Summary
================================================================================

This repository contains an iOS calorie-tracking app built with SwiftUI, plus a
WidgetKit extension and Firebase Functions backend.

Primary app goals:
- Log foods and nutrient totals throughout the day.
- Compute daily intake targets from burned calories +/- deficit/surplus settings.
- Pull/estimate nutrition from multiple sources:
  - Manual entry
  - PCC (Nutrislice) dining menus
  - USDA backend search
  - Open Food Facts barcode lookup
  - AI text meal parsing
  - AI food photo parsing
  - AI plate portion estimation for menu items
- Integrate Health/steps/workouts for energy expenditure.
- Persist state locally + sync app state to iCloud (CloudKit private DB).
- Power home/lock-screen widgets via App Group snapshot sharing.

Core architectural shape:
- Single large stateful root (`ContentView.swift`) as orchestrator.
- Feature views are mostly stateless/presentational with bindings/callbacks.
- Data persistence primarily through `@AppStorage` String-backed JSON blobs.
- External IO via service types (`*Service.swift`) with async/await networking.
- Background refresh via app delegate fetch + Health observer updates.

4) Runtime Lifecycle + Entry Points
================================================================================

Main app entry file: `Calorie Tracker/Calorie_TrackerApp.swift`

Important startup behavior:
1. `@main Calorie_TrackerApp` launches `ContentView` in dark mode preference.
2. `AppDelegate` registers for remote notifications.
3. Enables minimum background fetch interval.
4. Starts `HealthKitBackgroundObserver` (Health sample observers).
5. Kicks one background snapshot refresh task at launch.

Background refresh pathways:
- `application(_:performFetchWithCompletionHandler:)`
  concurrently preloads menus and refreshes widget snapshot.
- `application(_:didReceiveRemoteNotification:fetchCompletionHandler:)`
  listens for CloudKit app-state-change notification and posts
  `.cloudKitAppStateDidChange` in-process.

5) Root Orchestration (`ContentView.swift`)

`ContentView.swift` (~8k lines) is the app's central state machine.

5.1 Embedded systems inside ContentView file
- `AppCloudSyncService` (actor)
  - CloudKit record fetch/save for serialized app state payload.
  - Private DB record ID: `user-state`, record type `AppState`.
  - Stores payload via CKAsset JSON + numeric `updatedAt` field.
  - Ensures private DB subscription for silent push updates.
- `AppMenuPreloadService` (actor)

5.2 Major app state classes in ContentView

- Volatile UI state (`@State`) for tabs, sheets, loaders, selection, keyboard,

6) Domain Model Layer (`Models.swift`)
================================================================================

`MealGroup`
- Enum: breakfast/lunch/dinner/snack
- Contains display title and log sort rank.

`BMRSex`, `BMRProfile`
- Sex enum + body profile struct used for BMR calculations.
- `BMRProfile.isComplete` gate.

`MealEntry`
- Primary logged food model.
- Fields: id, normalized name, calories, nutrientValues, loggedCount, createdAt,
  mealGroup.
- Defensive decode behavior:
  - clamps negatives
  - backfills protein from nutrient map
  - infers meal group by logged timestamp if missing.

`QuickAddFood`
- Reusable saved food template with serving baseline.
- Enforces minimum serving amount and normalized serving unit.

`ExerciseType`, `ExerciseEntry`
- Exercise models for manual + HealthKit-mapped workouts.
- Supports optional custom name/symbol and reclassified walking calories.

`NutrientDefinition`
- Metadata schema for nutrient UI/goal controls.

`CalibrationState`
- Stores weekly smart-adjustment state:
  - enabled flag
  - calorie offset
  - recent errors
  - run status metadata
  - data quality counters.

`HealthWeighIn*` models
- Keep per-day representative weight sample with sample metadata.

7) Service Layer (Network/Platform)
================================================================================

7.1 Menu services
- `NutrisliceMenuService.swift`
  - Defines `DiningVenue` and `MenuItem/MenuLine/NutrisliceMenu`.
  - Venue-specific supported meal types and URL path rules.
  - Fetches daily menu from Nutrislice API endpoint.
  - Parses station headers + items into app structures.
  - Computes nutrient null-rate statistics across returned week.
  - Menu type schedule heuristic:
    - before ~10:45 -> breakfast
    - before 16:00 -> lunch
    - else dinner

7.2 Food sources
- `USDAFoodService.swift`
  - Calls backend endpoint `searchUSDAFoods`.
  - Backend base default:
    `https://us-central1-calorie-tracker-364e3.cloudfunctions.net`
  - Query via GET `?query=...`, maps/normalizes serving units.

- `OpenFoodFactsService.swift`
  - Directly calls OFF API v2 `/product/{barcode}`.
  - Parses dynamic nutriments and maps to internal nutrient keys.
  - Prefers serving values when serving metadata exists.

7.3 AI services
- `AITextMealService.swift`
  - POST `analyzeFoodText` with `mealText`.
- `AIFoodPhotoService.swift`
  - POST `analyzeFoodPhoto` with base64 + mimeType.
- `GeminiPlateEstimateService.swift`
  - POST `estimatePlatePortions` with image + selected menu items.
  - Expects `ozByFoodName`, `countByFoodName`, `baseOzByFoodName`, `rawText`.

7.4 Activity + health services
- `StepActivityService.swift`
  - CoreMotion pedometer integration (query + live updates).
  - Estimates active calories from distance * weight * coefficient.
  - If no distance, estimates from stride length * step count.

- `HealthKitService.swift`
  - Requests/read-only Health data authorization.
  - Pulls profile (sex, DoB, height, body mass).
  - Pulls today's workouts and reduced daily body-mass history.
  - Publishes authorization status + profile + workouts + errors.

- `ExerciseCalorieService.swift`
  - MET and distance-economy based exercise calorie estimation.
  - Running includes walking-equivalent overlap logic so walking burn from step
    model is not double-counted.

- `HealthKitWorkoutMapper.swift`
  - Maps many HK activity types to app presentation + `ExerciseType` buckets.
  - Chooses calories from HK or estimated depending on activity class.

7.5 Background/widget bridge
- `BackgroundWidgetRefreshService.swift`
  - Recomputes widget snapshot from UserDefaults + Health + steps.
  - Persists daily goal/burned archives.
  - Writes snapshot into App Group via `WidgetSnapshotStore`.
- `HealthKitBackgroundObserver` inside same file
  - Registers HK observer queries and triggers snapshot refresh on updates.

10) Cloud + Sync Behavior
================================================================================

Cloud state synchronization:
- Encodes local app state into `CloudSyncPayload`.
- Wraps with `CloudSyncEnvelope(updatedAt, payload)`.
- Saves/loads as CKAsset JSON in user private CloudKit DB.
- Establishes DB subscription for change notifications.
- App delegate routes silent pushes into in-app notification.
- ContentView gates applying payload with conflict protection flags
  (`hasBootstrappedCloudSync`, `isApplyingCloudSync`, upload tasks).

Menu preloading/cache:
- Actor `AppMenuPreloadService` iterates all venues + menu types.
- Fetches only when cache is empty or signature changed.
- Persists back into string-backed JSON defaults.

================================================================================

Source of truth: `NutrientCatalog.swift`
- Provides known nutrient metadata:
  key, display name, unit, default goal, min/max, step.
- Defines preferred display order.
- Defines non-trackable keys (`calories`, `re_vitamin_a`).
- Has default exclusions for consistently-null menu nutrients.
- Accepts imported nutrient values only for known keys and clamps >= 0.
- Fallback parser can generate unknown nutrient definitions from token patterns.

12) Activity + Calorie Model Details
================================================================================

Daily burned calories concept combines:
- BMR (from Health profile or fallback average).
- Exercise calories (manual + Health workouts).
- Step-based activity calories (walking model).
- Optional calibration offset (`CalibrationState.calibrationOffsetCalories`).

Then daily goal depends on goal type:
- deficit: goal = burned - deficitAmount
- surplus: goal = burned + surplusAmount

Weekend override:
- Optional alternate deficit/surplus value for Saturday/Sunday.

Double-count mitigation:
- Running workouts have `reclassifiedWalkingCalories` deducted from step-derived
  activity calories up to available amount.

17) Practical Mental Model for Future Changes
================================================================================

When changing behavior, think in this order:
1. Does this belong in pure data/model (`Models`, `NutrientCatalog`) or in UI?
2. If it crosses network boundaries, add/modify a dedicated service file.
3. Update `ContentView` only for orchestration/wiring/state transitions.
4. Persisted field changes require:
   - default initialization
   - encode/decode migration handling
   - cloud sync payload updates if needed
   - widget refresh impact review
5. For calories/burned logic changes, verify interaction between:
   - BMR
   - step calories
   - exercise calories
   - reclassified walking overlap
   - calibration offset
6. If UI is in menu/quick-add/plate flows, validate using UITests and by checking
   search + keyboard + bottom-bar overlap behavior.
